# Настройка окружения — что нужно сделать руками

Этот файл — чеклист разовых шагов, которые нужно выполнить в веб-интерфейсах (Supabase,
Google), потому что я не могу создавать аккаунты или проходить регистрацию за вас. Всё
остальное (код, схема БД, скраперы) делается прямо в Claude Code.

## 1. Supabase — проект приложения (операционная БД)

Раз аккаунт Supabase уже создан:

1. На https://supabase.com/dashboard → **New project**.
2. Название проекта: например `pricecompare-app`.
3. **Region: выбрать регион в ЕС** (например Frankfurt/Ireland) — важно для GDPR, чтобы данные
   пользователей физически хранились в Евросоюзе (см. `legal/PRIVACY_POLICY.md` §4).
4. Задать надёжный database password (Supabase сам предложит сгенерировать) — сохраните его в
   менеджере паролей, он понадобится для прямого доступа к Postgres, если понадобится.
5. После создания проекта — зайти в **SQL Editor** и выполнить содержимое файла
   [`supabase/migrations/0001_init_schema.sql`](supabase/migrations/0001_init_schema.sql) —
   это создаст все таблицы, индексы и политики доступа (RLS) из модели данных SPEC.md §9.
6. В разделе **Authentication → Providers** — убедиться, что Email-провайдер включён (для
   регистрации по email/паролю). Соцлогин (Google и т.п.) можно подключить позже, не блокирует
   MVP.
7. В разделе **Project Settings → API** — скопировать:
   - **Project URL**
   - **anon public key**

   Эти два значения не секретны в классическом смысле (anon-ключ рассчитан на использование
   прямо в клиентском приложении — вся защита данных обеспечивается через RLS-политики из
   миграции), но всё равно не стоит выкладывать их в публичный git-репозиторий большими буквами
   в коммитах истории — передайте их мне текстом в чат, когда будете готовы подключать
   приложение к Supabase, и я подставлю их в конфиг приложения через `--dart-define` (не в
   коде, который попадёт в git).

## 5. GitHub — автоматический скрапинг по расписанию

Чтобы цены в приложении обновлялись сами (без ручного запуска скрапера и без новой сборки
APK), скрапинг переезжает на GitHub Actions — бесплатный сервис, который сам по расписанию
запускает `scraper/*.py` и пишет результат в Supabase. Файл workflow уже готов:
[`.github/workflows/scrape.yml`](.github/workflows/scrape.yml) (по умолчанию — раз в сутки,
можно менять частоту в файле). Это то, что мне для этого нужно от вас (создание аккаунтов и
секретов — то, чего я не могу сделать за вас):

1. Создать репозиторий на https://github.com/new (можно приватный — но см. примечание ниже про
   лимит бесплатных минут; для скрапинга публичность репозитория не создаёт рисков, ключи
   Supabase хранятся отдельно как секреты, не в коде).
2. Прислать мне ссылку на репозиторий (или самостоятельно выполнить `git remote add origin ...`
   и `git push`) — я подготовлю коммит с текущим кодом.
3. В репозитории: **Settings → Secrets and variables → Actions → New repository secret** —
   добавить 4 секрета (значения — из `scraper/.env`, который у вас уже есть локально и никогда
   не попадает в git):
   - `SUPABASE_APP_URL`
   - `SUPABASE_APP_SERVICE_KEY`
   - `SUPABASE_ARCHIVE_URL`
   - `SUPABASE_ARCHIVE_SERVICE_KEY`
4. Проверить, что Actions включены (обычно включены по умолчанию) — вкладка **Actions** в
   репозитории, там же можно запустить workflow вручную кнопкой **Run workflow**, не дожидаясь
   расписания.

**Примечание про лимит:** полный обход всех рубрик обоих магазинов занимает по времени около
часа. У приватных репозиториев на бесплатном плане GitHub — 2000 минут Actions в месяц; при
ежедневном запуске это может почти исчерпать лимит. У публичных репозиториев Actions бесплатны
без ограничения по минутам. Если репозиторий приватный и лимита не хватает — можно снизить
частоту (например, раз в 2–3 дня) прямо в `cron` внутри `scrape.yml`.

Позже (по вашим словам — "в перспективе") можно будет перенести этот же скрипт на свой
сервер/VPS с обычным cron — тогда лимиты и расписание GitHub Actions будут не нужны, а сам
скрапер и `match_products.py` менять не придётся, поменяется только то, что их запускает.
