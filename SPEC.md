# PriceCompare LV — начальная спецификация продукта (v0.1)

> Рабочее название приложения условное. Заменить, когда появится бренд.

## 1. Видение продукта

Мобильное приложение (Android → iOS) для сравнения цен на продукты питания в разных сетях
магазинов Латвии. Пользователь открывает приложение, ищет товары или категории, видит, где
дешевле, собирает список покупок и получает чёткий ответ: **в каком магазине (или наборе
магазинов) купить весь список дешевле всего** — включая сравнение по цене за единицу измерения
(€/кг, €/л, €/шт), а не только по цене за упаковку.

Ключевая ценность — **наглядная, посчитанная экономия**, а не просто список цен.

## 2. Целевая аудитория

- Жители Латвии со средним и низким доходом, для кого экономия на еде — ощутимый фактор.
- Люди, которые хотят покупать привычные продукты, но по минимальной цене из доступных вариантов.
- Не «гурманы акций», а рациональные покупатели: им нужен быстрый ответ «куда идти», а не
  бесконечный список акций для просмотра.

## 3. Рынки и локализация

- **Первый рынок:** Латвия. Архитектура данных должна закладывать мультирынок с первого дня
  (страна/регион как поле у сети магазинов и у прайс-листа), даже если UI на старте показывает
  только Латвию.
- **Интерфейс:** 3 языка сразу — латышский (основной), русский, английский.
- **Данные о товарах:** названия товаров хранятся на языке магазина-источника (обычно
  латышский/русский вперемешку в зависимости от сети); для UI-категорий и фильтров используется
  собственный переведённый справочник категорий (см. §9.2), не зависящий от языка магазина.

## 4. Платформа и технологический стек

| Слой | Выбор | Причина |
|---|---|---|
| Мобильное приложение | **Flutter (Dart)** | Один код на Android и iOS, бесплатный SDK, хорошая производительность для списков/фильтров, богатая экосистема виджетов |
| Backend / БД / Auth | **Supabase** (Postgres + Auth + Storage + Edge Functions), бесплатный tier на старте | Даёт БД, аутентификацию и REST/Realtime API «из коробки» без написания своего backend-сервера; бесплатного плана достаточно для MVP; при росте — платный tier или миграция на self-hosted Postgres без переписывания логики (это тот же Postgres) |
| Скрапинг цен | Отдельные скрипты (Python, `httpx`/`playwright` + `BeautifulSoup`) по расписанию | Скрапинг — независимый от приложения процесс; можно гонять на бесплатных раннерах (например, GitHub Actions по cron) или на дешёвом VPS, если понадобится больше контроля/стабильности (сайты с JS-рендерингом, антибот) |
| Реклама | Google AdMob | Стандартный, бесплатный, встроенная поддержка в Flutter, монетизация баннерами с первого дня |

**Среда разработки:** пишем и итерируем код прямо здесь, в Claude Code — редактирование файлов,

## 5. Основные функции (MVP)

1. **Каталог товаров по категориям** — сразу весь ассортимент продуктов питания (не только
   базовый набор категорий), иерархия категория → подкатегория, по образцу структуры,
   которую уже используют сами магазины (см. §8.0: у Rimi и Barbora готовая, детальная
   иерархия категорий, которую можно взять за основу собственного справочника).
2. **Поиск**:
   - по названию/бренду («Kraft кетчуп», «макароны Barilla»);
   - по категории (просто «кетчуп» → список всех кетчупов всех сетей).
3. **Сравнение цен** для найденного товара/категории:
   - сортировка по цене за упаковку;
   - сортировка по цене за единицу измерения (€/кг, €/л, €/шт) — именно это решает кейс
     «пол-упаковки дороже за кг, чем целая упаковка»;
   - фильтры: магазин, ценовой диапазон, бренд, только акции.
4. **Список покупок (корзина)**:
   - пользователь добавляет товары в список;
   - приложение считает: **(а)** в каком ОДНОМ магазине купить весь список дешевле всего
     (сумма), **(б)** по кнопке — альтернативный экран с оптимальным распределением списка по
     нескольким магазинам с показом итоговой экономии в € по сравнению с вариантом (а).
5. **Каталог акций/буклетов** сети — отдельная лента акций по выбранным магазинам.
6. **Аккаунт пользователя** (регистрация/логин через Supabase Auth, email или соцсеть):
   - сохранённые списки покупок;
   - избранные магазины;
   - история сравнений (не критично для v1, но заложить в модель данных).
7. **Баннерная реклама** (AdMob) — ненавязчиво, не перекрывает список цен.
8. **Атрибуция источников данных** (обязательно для MVP, см. §8.0): на экране сравнения цен —
   пометка источника у каждой цены («по данным Barbora» для Maxima, аналогично для Rimi), плюс
   общий дисклеймер «Цены собраны из открытых источников и могут отличаться от актуальных цен
   в конкретном магазине». Не косметика, а требование к доверию пользователей и к юридической
   защите приложения.

### Не входит в MVP (роадмап v2+)
- Учёт локации/расстояния до магазина (нужны координаты филиалов).
- Push-уведомления о новых акциях на отслеживаемые товары.
- iOS-сборка (архитектура готова, публикация — отдельный этап).
- Другие страны/рынки.
- Партнёрские ссылки на онлайн-заказ (Barbora и т.п.) — возможное будущее направление
  монетизации, если сети предложат партнёрку.

## 6. Пользовательские сценарии (флоу)

**Сценарий А — быстрый поиск:**
Открыл приложение → ввёл «кетчуп» → увидел список кетчупов всех сетей, отсортированный по
€/кг → выбрал самый выгодный → увидел, в каком магазине.

**Сценарий Б — сборка списка на неделю:**
Открыл список покупок → добавил 10-15 позиций из каталога/поиска → нажал «Сравнить» →
увидел: «Дешевле всего купить всё в Rimi — €23.40» и кнопку «А если по разным магазинам?» →
там: «Rimi + Maxima — €21.10, экономия €2.30, но 2 магазина вместо одного».

**Сценарий В — акции:**
Зашёл в ленту акций выбранной сети → посмотрел, что сейчас по скидке → добавил нужное в
список покупок.

## 7. Логика сравнения и ранжирования — детали

- Базовая единица сравнения — **цена за нормализованную единицу** (кг / л / шт), вычисляется
  из цены упаковки и её объёма/веса, которые нужно извлекать из данных магазина при скрапинге
  (частично — парсингом текста «500 г», «1 л», частично — вручную/эвристиками, если магазин не
  даёт вес машиночитаемо).
- Сортировки: по цене за упаковку, по цене за единицу, по разнице с обычной ценой (акции).
- Для корзины: алгоритм (а) — тривиальная сумма цен по каждому магазину, минимум;
  алгоритм (б) — для каждой позиции выбирается магазин с минимальной ценой (простой greedy),
  с отображением итоговой суммы и количества магазинов. Более сложная оптимизация (например,
  ограничение «не больше 2 магазинов») — можно добавить после MVP, если понадобится.

## 8. Стратегия сбора данных (скрапинг)

### 8.0 Результаты разведки сайтов (проверено вживую, 2026-07-18)

**Rimi (rimi.lv → раздел e-veikals):**
- Полноценный интернет-магазин со всем ассортиментом, категориями, поиском, ценами и ценой за
  единицу (€/кг, €/л) — всё видно **без логина и без выбора адреса доставки**.
- Данные карточек товаров отдаются сразу в HTML (server-side rendering) — цена, вес/объём
  упаковки, цена за единицу измерения уже посчитана и присутствует в тексте страницы. Значит,
  скрапинг = обычный HTTP GET + разбор HTML, без необходимости выполнять JS.
- Отдельно нашёлся чистый JSON-эндпоинт для дерева категорий:
  `GET /e-veikals/api/v1/content/category-tree?locale=lv` — отдаёт полную иерархию категорий
  с человекочитаемыми URL вида `/e-veikals/lv/produkti/.../c/SH-2-1-3`. Это готовая основа для
  построения нашего собственного каталога категорий.
- **robots.txt** (`rimi.lv/robots.txt`) разрешает обход, блокирует только `/admin/`,
  `/search?q=*`, `/checkout`, `/account`, `/cart/` — то есть страницы категорий и товаров
  явно не запрещены. Есть sitemap.xml.

**Rimi — акции, отдельная проверка (2026-07-18):**
- Страница `/e-veikals/lv/akcijas-piedavajumi` (внутри e-veikals) — это **та же чистая SSR
  HTML-структура**, что и обычные страницы категорий: обычная цена, цена со скидкой, цена за
  единицу измерения — всё в тексте страницы, без PDF и без JS-рендеринга. Более того, там же
  присутствует **«30 dienu zemākā cena» («30-дневная минимальная цена»)** — обязательный по
  директиве ЕС (Omnibus Directive) показатель, который сама Rimi уже считает и публикует, чтобы
  показать, что скидка не фиктивная. Это готовый, чистый источник данных для раздела акций в
  приложении — парсится так же просто, как обычный каталог, отдельного решения не требует.
- Страница `/akcijas-un-bukleti` (на корпоративном rimi.lv, не в e-veikals) — это, наоборот,
  просто список ссылок на цифровые буклеты-«газеты», которые открываются на **отдельном
  поддомене `bukleti.rimi.lv`** — это специализированный интерактивный flipbook-просмотрщик
  буклетов (по характеру похож на типовые сервисы просмотра рекламных буклетов, разворот в виде
  журнала). В ходе проверки этот поддомен не удалось полноценно загрузить в разумное время
  (тяжёлый интерактивный вьюер) — по характеру такие вьюеры обычно рендерят страницы как
  картинки/canvas, а не как текст, то есть парсинг сложнее (нужен либо разбор внутреннего
  API вьюера, если он есть, либо OCR картинок страниц буклета).
- **Вывод:** для MVP буклеты `bukleti.rimi.lv` не нужны — раздел «Акции» полностью закрывается
  страницей `/e-veikals/lv/akcijas-piedavajumi`, которая и проще в парсинге, и даёт более
  структурированные данные (включая 30-дневную минимальную цену). Буклеты можно оставить на
  будущее как отдельную, необязательную задачу, если понадобятся акции, которых нет в
  e-veikals (например, товары, продающиеся только оффлайн в магазине).

**Maxima (maxima.lv) — важная поправка в план:**
- Сайт maxima.lv на деле **не является интернет-магазином с полным каталогом** — это
  корпоративный/маркетинговый сайт с разделом еженедельных акций («Piedāvājumi»), без поиска
  по всему ассортименту и без каталога цен товар-за-товаром.
- Сайт защищён Cloudflare bot-challenge (обнаружены вызовы `cdn-cgi/challenge-platform`) —
  технически затрудняет автоматический скрапинг.
- **robots.txt maxima.lv явно запрещает доступ ИИ-краулерам**, включая `ClaudeBot`, `GPTBot`,
  `Google-Extended`, `CCBot`, `Bytespider`, `Amazonbot`, `meta-externalagent` (при этом для
  обычного `User-agent: *` стоит `Allow: /` с пометкой `Content-Signal: ai-train=no`). Из-за
  этого явного запрета для ClaudeBot дальнейший сбор данных с maxima.lv в этой сессии не
  проводился.
- **Настоящий интернет-магазин с полным ассортиментом и ценами Maxima работает на отдельном
  сайте — barbora.lv** (в футере указан оператор SIA «Patrika»; ссылка на Barbora есть в шапке
  maxima.lv). Именно там нужно брать данные, если цель — покрыть ассортимент и цены Maxima.

**Barbora (barbora.lv) — практическая альтернатива для данных Maxima:**
- Полноценный каталог: категории с человекочитаемыми ЧПУ-адресами
  (`/piena-produkti-un-olas/piens/pasterizets-piens` и т.п.), цены и цена за единицу
  показываются сразу на карточке товара без логина.
- Обнаружен чистый REST JSON API: `/api/eshop/v1/...` (например
  `category/offertree`, `product/getSearchPlaceholderInventories`,
  `recommendations/GetHomeRecommendationsInventories`) — версионируется (`api-1.53.0` видно в
  футере страницы), похоже на стабильный, поддерживаемый API, а не внутренний служебный.
- **robots.txt** барборы не содержит никаких ограничений для ИИ-ботов и вообще для обычного
  скрапинга — блокирует только служебные разделы (корзина/оформление/личный кабинет/доставка
  на разных языках сайта).

**Решение (подтверждено):** для MVP парсим **Rimi (rimi.lv/e-veikals) напрямую** и **Barbora
(barbora.lv) как источник данных по ассортименту и ценам Maxima**, а не maxima.lv.

**Требование к отображению в приложении (важно, продуктовое и юридическое):**
- Явно указывать источник данных для каждой цены/сети. Для позиций из Barbora — подпись вида
  «Цены указаны по данным Barbora» и/или совместное отображение логотипов Maxima + Barbora
  (уточнить у юриста допустимость использования чужих логотипов; как минимум — текстовая
  атрибуция обязательна с первого дня).
- Причина: цены в физическом магазине Maxima и на сайте Barbora **на практике могут
  отличаться** — приложение не должно создавать впечатление, что это гарантированная цена в
  кассе конкретного магазина Maxima.
- Общий дисклеймер уровня приложения (не только по Maxima/Barbora, а везде): «Информация о
  ценах собрана из открытых источников (публичные страницы интернет-магазинов) и может
  отличаться от актуальной цены в конкретном магазине». Разместить в: (а) экране сравнения цен
  рядом с ценой/списком, (б) отдельном экране «О приложении», (в) политике конфиденциальности
  / пользовательском соглашении.
- Используем только открытую, публично доступную информацию (см. §8.0 про robots.txt) — без
  логина, без приватных API, без обхода технических ограничений доступа.

### 8.1 Общий подход к скрапингу

- Подход: скрапинг открытых страниц интернет-магазина (и/или PDF-буклетов акций, где это
  применимо) → нормализация → запись в БД. Ничего приватного не запрашиваем, только публично
  доступные страницы, в рамках правил robots.txt каждого сайта.
- Скрипты по расписанию (например, раз в сутки) обновляют цены; хранится история цен (для
  будущего «график цены» и обнаружения фейковых «скидок»).
- **Важно (риск, см. §12):** при появлении официального API/фида от сети — переходить на него
  в приоритете, так как это стабильнее скрапинга, который ломается при редизайне сайта.

### 8.2 Проблема сопоставления «один и тот же товар в разных магазинах»
Это самый сложный технический момент проекта: у Rimi и Barbora разные внутренние ID товаров,
разные названия и фото одного и того же продукта. Предлагаемый подход для MVP:
- Сопоставление по **штрихкоду (EAN/GTIN)**, если он присутствует в данных сайта — самый
  надёжный способ, используется как первичный ключ товара во внутреннем каталоге.
- Если штрихкода нет — fuzzy-сопоставление по **(бренд + нормализованное название + вес/объём)**
  с ручной проверкой найденных совпадений на первых порах (пока каталог небольшой — 2 сети).
- Товары, которые не удалось сопоставить, всё равно живут в каталоге как отдельные позиции —
  просто не участвуют в кросс-магазинном сравнении, пока не сматчены.

## 9. Модель данных (черновая)

- **stores** — сеть/филиал (name, country, logo, website).
- **categories** — иерархия категорий (собственная, не зависящая от магазина).
- **products** — канонический товар (ean, canonical_name, brand, category_id, unit_type
  [kg/l/pcs], unit_size).
- **store_products** — конкретный товар в конкретном магазине (product_id, store_id,
  store_sku, raw_name, package_price, package_size, scraped_at).
- **price_history** — краткая, рабочая история цены товара для нужд самого приложения
  (детекция акций, «было/стало» на карточке товара). Хранится в основной (операционной) БД,
  ретеншн можно ограничить (например, последние N месяцев) — см. §9.1 про отдельный
  долгосрочный архив, который эту роль не выполняет и не заменяет.
- **promotions** — акции (store_product_id, discount_price, valid_from, valid_to, source
  [сайт/буклет]).
- **users**, **shopping_lists**, **shopping_list_items**, **favorite_stores** — через
  Supabase Auth + свои таблицы.

### 9.1 Отдельная база для долгосрочной статистики цен (личные/научные задачи)

Отдельное требование, не связанное напрямую с функциональностью приложения для конечных
пользователей: вести **отдельную, независимую от операционной БД приложения базу данных**,
куда постоянно (при каждом скрапинге) записывается история цен **по всем товарным группам**
(не только по товарам, сматченным между магазинами, а вообще по всему, что было спарсено) —
для последующего личного анализа и научных задач (например, отслеживание продуктовой
инфляции в Латвии за длительный период, сезонность цен, паттерны акций по категориям).

**Почему отдельная база, а не просто таблица в общем проекте:**
- Разная природа нагрузки: операционная БД приложения — частые лёгкие запросы (пользователи
  листают каталог); архив цен — редкие, но объёмные записи при каждом скрапинге и объёмные
  выборки при анализе (например, «дай все цены по категории "молоко" за 2 года»). Смешивать
  это в одной БД неэффективно и рискованно (тяжёлые аналитические запросы могут замедлять
  работу самого приложения).
- Разный жизненный цикл: данные приложения могут ротироваться/удаляться (например, при
  удалении аккаунта пользователя, см. `legal/ACCOUNT_DELETION_POLICY.md`); архив цен, наоборот,
  должен копиться **бессрочно** и никогда не тронут очисткой, привязанной к пользователям —
  это не персональные данные вообще, только цены товаров.
- Разные требования к экспорту: для научной работы удобнее иметь возможность выгрузить данные
  как есть в CSV/Parquet для анализа в Python/R/Excel, не трогая продовую инфраструктуру
  приложения.

**Предлагаемая структура архива (черновая):**
- `price_observations` — append-only лог: `store_id`, `raw_product_key` (SKU/URL магазина,
  устойчивый идентификатор даже если товар не сматчен с каноническим), `category_path` (как
  на сайте магазина, без привязки к нашей собственной таксономии — это тоже интересно для
  анализа расхождений), `raw_name`, `package_size`, `price`, `unit_price`, `is_promo`,
  `observed_at`.
- Для экономии места — писать новую строку **только при изменении цены** (модель
  «изменение-событие», SCD-2 стиль: `price`, `valid_from`, `valid_to`), а не при каждом
  ежедневном скрапинге, если цена не поменялась. Это резко снижает объём данных за годы,
  сохраняя полную историю изменений.
- Хранить и товары, которые не удалось сопоставить между магазинами (см. §8.2) — для научной
  задачи важна вся картина по рынку, а не только кросс-магазинные совпадения.

**Технически (решено):** на старте — отдельный бесплатный Supabase-проект (или отдельная
схема в Postgres), независимый от проекта приложения, как предложено выше. Это временное
решение: план — постепенно поднять собственный VPS и перенести туда архив (и, возможно, сам
скрапер) по мере роста объёма данных и по мере готовности. Причины переезда на свой VPS
именно для архива (а не обязательно для всего проекта сразу):
- бесплатные лимиты Supabase (место в БД, трафик) рассчитаны на приложение, а не на
  многолетний научный архив с ежедневными записями по всему ассортименту — это первое, что
  упрётся в лимит;
- на своём VPS нет ограничений по объёму хранения (кроме диска сервера) и по количеству
  проектов/баз — можно спокойно наращивать архив, ставить свою СУБД, делать любые
  аналитические выборки без оглядки на тарифные ограничения хостинг-платформы;
- операционная БД приложения (Supabase) может остаться как есть — миграция архива никак не
  затрагивает работу самого приложения для пользователей, если с самого начала архив физически
  отделён (см. причины выше).

Практический план: начать с Supabase (второй проект) для архива уже сейчас, чтобы не
блокировать разработку приложения; когда VPS будет готов — перенести таблицу
`price_observations` целиком (это простой дамп/restore для Postgres, так как формат данных не
меняется) и переключить скрапер на запись в новую БД. Если объём вырастет ещё сильнее —
дополнительно рассмотреть периодическую выгрузку старых данных в файлы (Parquet/CSV) на диске
VPS вместо хранения всего в живой БД. Схему с самого начала проектировать с расчётом на
многолетнее хранение (см. структуру выше), чтобы миграция на VPS была простым переносом
данных, без переделки модели.

**Юридический аспект:** это не персональные данные пользователей приложения, а публичные
данные о ценах товаров — требования GDPR (см. `legal/PRIVACY_POLICY.md`) на этот архив не
распространяются. Отдельно стоит убедиться, что использование этих данных в личных/научных
целях не противоречит условиям использования сайтов-источников (см. §8.0 про robots.txt) —
поскольку данные собираются в рамках того же скрапинга, что и для нужд приложения, отдельного
дополнительного риска это не создаёт.

## 11. Юридические и технические риски (требуют внимания по ходу разработки)

- **Скрапинг:** по факту проверки (см. §8.0) — Rimi и Barbora разрешают обход в robots.txt и
  не блокируют страницы категорий/товаров; **maxima.lv отдельно и явно запрещает доступ

  bot-challenge — по этой причине предложено брать данные Maxima через barbora.lv, а не с
  самого maxima.lv. В любом случае: не создавать чрезмерную нагрузку (частота запросов),
  хранить только публично доступные данные о ценах, без персональных данных пользователей
  магазинов.
- **GDPR:** раз в MVP уже есть аккаунты пользователей — нужна политика конфиденциальности,
  согласие на обработку персональных данных, право на удаление аккаунта/данных.
- **iOS:** нужен платный Apple Developer аккаунт (99 $/год) для публикации в App Store —
  это единственная неизбежная статья расходов, когда дойдём до iOS-релиза; для разработки/
  тестирования на симуляторе он не обязателен.
- **AdMob / Google Play:** Google Play Developer аккаунт уже есть — публикация Android-версии
  ничем не блокируется с этой стороны.

## 14. Дизайн и визуальный стиль

Требование: лёгкий, понятный интерфейс; цвета, вызывающие **доверие** и ощущение **выгоды**,
приятные глазу, без визуального шума — приложение должно «читаться» мгновенно (это же продукт
про экономию времени и денег, интерфейс не должен требовать усилий).

### 14.1 Цветовая палитра (предложение)

| Роль | Цвет | Почему |
|---|---|---|
| **Основной (primary)** | Тёмный сине-бирюзовый, например `#0F6E73` / `#1B6FA8` | Синий/сине-зелёный — классический «цвет доверия» в финтехе и сравнении цен: спокойный, надёжный, не давит |
| **Акцент «выгода»** | Насыщенный зелёный, например `#2E9E5B` | Зелёный = деньги, экономия, «да, бери». Используется для бейджей «Дешевле всего», сумм экономии, индикатора самой выгодной цены |
| **CTA / «лучшее предложение»** | Тёплый янтарный/коралловый, например `#F5A623` | Отдельный акцент для кнопок действия и хайлайта акций — создаёт ощущение «интересной находки», не тревожный, в отличие от красного |
| **Фон** | Почти белый, `#FAFAFA`–`#F7F9F9` | Чистота, лёгкость, не устаёт глаз при долгом скроллинге списков цен |
| **Текст** | Тёмно-серый/графитовый, `#1F2933`, не чистый чёрный | Мягче для глаз, чем #000, при этом высокий контраст для читаемости цифр цен |
| **Предупреждение/рост цены** | Приглушённый красный, `#D64545`, **точечно** | Только для явно негативных сигналов (цена выросла) — не как основной цвет, чтобы не создавать тревожность во всём интерфейсе |

Важно: не полагаться только на цвет для передачи смысла (зелёное/красное) — рядом всегда
иконка/стрелка/подпись, чтобы было понятно и людям с нарушениями цветовосприятия. Контраст
текста и фона — по рекомендациям WCAG AA как минимум.

### 14.2 Типографика и общий стиль

- Шрифт: чистый, читаемый sans-serif с чуть смягчёнными формами (например, Inter, Nunito,
  или системный шрифт Roboto на Android по умолчанию — не тратить время/бюджет на кастомный
  шрифт для MVP, если системный решает задачу).
- Крупные, жирные цифры цен и суммы экономии — это главный визуальный акцент на экране, всё
  остальное вторично по весу.
- Карточки товаров: минимум текста, акцент на цену/цену-за-единицу/бейдж выгоды, фото товара
  (если есть с сайта источника) — насыщенность считывается за секунду, без чтения описаний.
- Много воздуха (padding/spacing) — не перегружать экран, особенно с учётом того, что списки
  цен по своей природе длинные (десятки позиций при сравнении категории).

### 14.3 Практический подход к реализации

Для экономии времени и денег на разработке — не создавать отдельный дизайн-макет в Figma для
MVP, а сразу описать эту палитру как **Material 3 `ColorScheme.fromSeed`** в самом Flutter-
коде (Flutter/Material 3 сам генерирует согласованные светлую и тёмную темы из одного seed-
цвета, включая контрастные варианты для текста/поверхностей). Это позволяет сразу увидеть
живой интерфейс на реальных экранах вместо статичных макетов, и быстро донастроить акцентные
цвета (зелёный для выгоды, янтарный для CTA) поверх сгенерированной темы.

### Этап 1 — инициализация проекта (могу сделать в Claude Code)

### Этап 3 — соединяем приложение с реальными данными
13. Подключить Flutter-приложение к Supabase: аутентификация, каталог, поиск, сравнение цен,

15. Добавить атрибуцию источников на экране сравнения («по данным Barbora» и т.п., см. §8.0) и
    общий дисклеймер об открытых источниках.

### Этап 4 — тестирование и первый релиз
16. Ручное тестирование на реальном Android-устройстве (USB + `flutter run`).
17. Разместить публично тексты из `legal/` (Google Play требует публичный URL политики
